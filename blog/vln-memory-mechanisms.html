VLN的记忆革命：2025年最新论文告诉你机器人如何「记」路

2026-03-25 · 技术博客 · VLN, 具身导航, 记忆机制, LLM, Transformer
🧠
导航机器人如何记住走过的路？

导航机器人最大的问题不是「看不见」，而是「记不住」。2025年的三篇顶会论文从三个不同角度，彻底重新设计了VLN的记忆机制。

为什么「记忆」是VLN的核心问题

视觉语言导航（Vision-and-Language Navigation，VLN）的任务听起来很直觉：给机器人一句话指令，让它在3D环境里走到目标点。但实际做起来，有个让所有人头疼的问题——机器人走了5步之后，怎么记得自己从哪来、看见了什么、指令里的哪些词还没被满足？

传统方法用历史观察帧拼成时序序列，喂给Transformer。帧一多，计算量爆炸；帧一少，上下文丢失。这个矛盾一直没有好解法。

2025年，三篇论文从不同角度给出了答案：MapNav（ACL'25） 用语义地图替代帧序列，COSMO（ICCV'25） 用状态空间模型选择性压缩记忆，NavRAG（ACL'25 Findings） 则从数据层面重新定义了"导航指令"本身。

MapNav：把记忆变成一张地图

论文：MapNav: A Novel Memory Representation via Annotated Semantic Maps for VLM-based VLN（ACL 2025）
来源：同济大学、北大、哈工大深圳等
arXiv：2502.13451

MapNav的核心洞察很简单：历史帧序列本质上是在记「我看到了什么」，但机器人真正需要的是「环境长什么样」。

做法：每个导航任务开始时，MapNav构建一张自上而下的2D语义地图（Annotated Semantic Map, ASM），然后在每个时间步动态更新这张地图。关键点在于，地图上不只有像素，还有文字标注——每个区域和关键物体都被打上文本标签，把抽象的视觉信息转化为VLM可以直接读取的结构化线索。

传统方法：obs_t1, obs_t2, obs_t3, ... obs_tn → [所有历史帧] → Transformer
MapNav：obs_t1~tn → [实时更新语义地图] → ASM → VLM决策

效果

  • 在模拟和真实环境都达到 SOTA
  • 显著降低存储和计算开销（不需要存所有历史帧）
  • 在未见过的新环境中泛化能力更强

我的看法：这个方法有点像人类导航的方式——我们不会记住走过的每一帧画面，而是在脑子里维护一张"草图地图"。把这个直觉工程化，并且让VLM直接读取地图上的文字标注，是个很优雅的设计。唯一的问题：在室外开放场景或动态环境中，实时维护一张准确的语义地图还是很有挑战的。

COSMO：选择性记忆，低成本高性能

论文：COSMO: Combination of Selective Memorization for Low-cost Vision-and-Language Navigation（ICCV 2025）
来源：同济大学
arXiv：2503.24065

COSMO的出发点是个现实问题：现在的VLN模型越来越大、越来越贵，但部署在机器人上需要轻量高效。怎么在不牺牲性能的前提下降低计算成本？

做法：COSMO是一个混合架构，把两种模块结合：
  1. 状态空间模型（SSM）模块：引入 RSS（Round Selective Scan）算子，在单次扫描中捕获所有token之间的全面关系。它的优点是可以选择性压缩不重要的历史信息，保留与当前指令最相关的视觉记忆。
  2. Transformer模块：用于精确的动作决策，处理已被SSM筛选过的紧凑上下文。
直白说：SSM负责「忘掉不重要的」，Transformer负责「用好记住的」。

为什么RSS有效

传统Mamba类SSM是线性扫描，处理序列时对空间关系的建模有限。RSS通过"环形扫描"策略，让每个token都能感知全局关系，同时保持线性计算复杂度。

效果

  • 低计算成本下达到业界领先性能
  • 在R2R等标准VLN基准上超越同量级模型
  • 模型体积显著小于纯Transformer方案

我的看法：COSMO的意义不只在于性能，更在于工程可行性。一个需要A100集群才能跑的导航模型，在真实机器人上没有意义。COSMO指出了一条轻量化的路：不是砍掉功能，而是让记忆更聪明地压缩。

NavRAG：从源头重新定义导航指令

论文：NavRAG: Generating User Demand Instructions for Embodied Navigation through Retrieval-Augmented LLM（ACL 2025 Findings）
作者：Zihan Wang, Yaohui Zhu, Gim Hee Lee, Yachun Fan
arXiv：2502.11142

NavRAG解决的问题更上游：VLN的训练数据本身就是错的。

现有数据的问题

现有VLN数据集（R2R、REVERIE等）的指令通常是"走到走廊、左转、进第三个门"这类步骤式指令。但真实用户是怎么说话的？"帮我找一下厨房的咖啡机"或者"我需要去那个有红沙发的房间"——这是需求式指令，完全不同。

用步骤式指令训练出来的模型，遇到真实用户会一脸懵。

NavRAG的解法

  1. 层次化场景描述树：用LLM对3D场景构建树结构，从全局布局（有几个房间、功能分区）到局部细节（桌上有什么），捕获完整的语义层次。
  2. 多角色用户模拟：模拟访客、维修工、导游等不同角色，每种角色有不同的需求和表达习惯。
  3. RAG式指令生成：根据角色需求，从场景树中检索相关信息，然后用LLM生成符合该角色沟通习惯的导航指令。

规模：在861个3D场景中生成了超过200万条导航指令。

效果

  • 用NavRAG数据训练的模型，在R2R/CVDN等标准基准上性能更高
  • 对LLM-based导航方法友好（纯LLM方法在NavRAG基准上表现更好）
  • 指令多样性和用户风格匹配度显著优于基线

我的看法：NavRAG的贡献其实是在问一个根本问题：我们在训练模型按什么样的指令走路？ 如果训练分布就是错的，模型再强也没用。RAG + 层次化场景理解 + 角色扮演，这个组合不仅用在VLN数据生成上，对其他具身任务的数据扩充也有借鉴意义。

给具身导航研究者的启示

  1. 记忆不必是帧序列：地图、图结构、压缩状态都是可行的替代形式，选什么取决于你的任务和计算约束。
  2. 数据质量比数量更重要：NavRAG证明了指令分布对模型性能的影响，比单纯堆叠数据量更关键。
  3. 轻量化是刚需，不是妥协：COSMO的成功说明轻量化和高性能可以兼得，关键在于架构设计而非参数堆砌。
  4. VLM是新的基础设施：这三篇论文都在不同层面使用了VLM，从记忆编码（MapNav）到数据生成（NavRAG），VLM已经成为VLN研究的基础设施而非创新点本身。
